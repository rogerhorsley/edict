### 3. 性能优化
**输入**：性能瓶颈、优化目标
**输出**：优化方案、实施结果

**流程**：
1. 性能分析（CPU、内存、I/O、网络）

4. 效果验证

**交付物**：
```markdown
## 性能优化报告：AI客服API

### 性能问题
**现状**：API响应时间P95为2.5秒，远超1秒目标

### 性能分析
**工具**：Python cProfile + AWS X-Ray

**瓶颈识别**：
1. **数据库慢查询**（占比60%）
   - `SELECT * FROM conversations WHERE user_id = ?`
   - 原因：未建索引，全表扫描
   - 耗时：1.5秒

2. **GPT-4 API延迟**（占比30%）
   - 平均延迟：0.8秒
   - 原因：每次请求都调用，无缓存

3. **JSON序列化慢**（占比10%）
   - 大对象序列化耗时：0.2秒

### 优化方案
**优化1：数据库索引**
```sql
CREATE INDEX idx_user_id ON conversations(user_id);
CREATE INDEX idx_session_id ON conversations(session_id);
```
**效果**：查询从1.5秒 → 0.05秒（提升30倍）

**优化2：GPT-4响应缓存**
```python
# 相同问题直接返回缓存
cache_key = f"gpt4:{hash(prompt)}"
if redis.exists(cache_key):
    return redis.get(cache_key)
```
**效果**：命中率30%，这部分请求从0.8秒 → 0.01秒

**优化3：JSON序列化优化**
```python
# 使用orjson替代标准json库
import orjson
response = orjson.dumps(data)
```
**效果**：序列化从0.2秒 → 0.05秒（提升4倍）

### 实施结果
| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| P50响应时间 | 1.8秒 | 0.4秒 | 78% ↓ |
| P95响应时间 | 2.5秒 | 0.9秒 | 64% ↓ |
| P99响应时间 | 3.2秒 | 1.5秒 | 53% ↓ |
| QPS | 30 | 80 | 167% ↑ |

**✅ 达标**：P95响应时间0.9秒，满足<1秒目标

### 后续优化计划
1. 增加数据库连接池（50→100）
2. 实施异步处理（长时间操作）
3. 增加边缘缓存（Cloudflare）
```

```markdown
## DevOps方案：AI客服系统

### CI/CD流程
```
代码提交 → GitHub
    ↓

    ↓

### 运维手册
**常见故障处理**：

**1. 服务宕机**
- 现象：健康检查失败
- 排查：检查日志、CPU、内存
- 处理：重启服务或回滚版本
- 预计恢复时间：5分钟

**2. 数据库慢查询**
- 现象：响应时间突然增加
- 排查：`SHOW PROCESSLIST`查看慢查询
- 处理：优化查询或增加索引
- 预计恢复时间：30分钟

**3. Redis连接耗尽**
- 现象：连接池报错
- 排查：检查连接数和慢操作
- 处理：增加连接池或优化Redis操作
- 预计恢复时间：15分钟
